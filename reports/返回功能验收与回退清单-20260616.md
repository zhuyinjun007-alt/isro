# 返回功能验收与回退清单

整理时间：2026-06-16  
用途：给下一轮返回功能修改建立固定门禁，避免再次把局部成功当成最终成功。

## 1. 适用范围

- 只针对 CSRO 的 `SRO_Client.exe` 和 `8-SR_GameServer.exe`。
- 不包含 DB、PK2、DLL、launcher、ShardManager、强化、界限突破石、防爆石、药化、过滤器等其他补丁。
- 不适用于“只做静态验证”的阶段；一旦写入字节，必须走完整回退流程。

## 2. 先决条件

1. 先冻结当前 live 状态。
2. 先记录当前 EXE/GS hash。
3. 先生成本轮独立备份。
4. 先写入 manifest。
5. 先确认本轮只改返回相关字节。

### 当前必须注意

- 现有报告里的 EXE hash 与本次只读复核读到的 hash 不一致，说明当前状态需要先重新冻结。
- 在没有重新冻结前，不能把旧报告 hash 当作最新事实。

## 3. 一轮修复的标准流程

| 步骤 | 动作 | 必要证据 | 失败后的动作 |
| --- | --- | --- | --- |
| 1 | 备份 GS / EXE | 备份路径、hash、时间戳 | 不进入写入 |
| 2 | 写入本轮补丁 | VA、file offset、bytes、manifest | 立刻停 |
| 3 | 静态检查 | 目标分支、handler、入口、旧按钮保留 | 若不通过，先判定是否可回滚 |
| 4 | 游戏内验证 | 第一次右键、第二次右键、旧按钮、新按钮 | 若失败，按本轮备份回滚 |
| 5 | 记录日志 | FatalLog、Dump、截图、时间点 | 若缺失，降级为证据不足 |
| 6 | 回滚确认 | 回滚后 hash 复核 | 必须回到本轮备份状态 |

## 4. 必测清单

### 4.1 入口与 UI

- 第一次右键返回卷轴，必须正常显示。
- 关闭后第二次右键返回卷轴，必须仍能正常显示。
- 不能再出现 UI 只有局部显示或点击即崩的情况。

### 4.2 旧功能

- `0xC8`
- `0xC9`
- `0xCA`
- `0xCB`

要求：

- 旧按钮必须仍走原 CSRO 链路。
- 不能因为新增按钮而破坏旧功能。

### 4.3 新功能

- `0xCC`：移动至队员位置
- `0xCF`：移动至使用者指定的位置

要求：

- `0xCC` 必须有组队门槛。
- `0xCF` 必须完整闭合到 GS 和 EXE。
- 两个按钮都不能只停留在静态分支或字符串层面。

### 4.4 服务端与副作用

- GS 收包必须可见。
- GS 校验必须可见。
- 传送结果必须可见。
- 失败时不能污染其他返回补丁。

## 5. 失败判定与回退规则

| 失败类型 | 判定 | 处理 |
| --- | --- | --- |
| UI 未出现 | 入口或分支不对 | 回滚本轮补丁 |
| 第二次打开崩溃 | 生命周期未闭合 | 回滚本轮补丁 |
| 旧功能失效 | 旧链路被污染 | 回滚本轮补丁 |
| `0xCC/0xCF` 不可用 | 新链路未闭合 | 回滚本轮补丁 |
| 客户端 dump | 运行时失败 | 先保留 dump，再回滚 |
| FatalLog 异常 | 运行时失败 | 先保留日志，再回滚 |
| 静态通过但游戏失败 | 静态证据不足以收口 | 以游戏内结果为准，回滚 |

## 7. 这轮应当保留的结论

- 静态通过只能作为准入，不是最终成功。
- 旧按钮必须保持原链路。
- 新按钮只能新增，不能覆盖旧按钮。
- 当前状态必须先重新冻结，再进入下一轮写入。

## 8. 最低动作建议

1. 先重新冻结当前 EXE / GS hash。
2. 先做旧按钮保留表的实证核对。
3. 先做 `0xCC/0xCF` 的最小新增闭合，不碰旧按钮。
4. 一旦游戏内失败，先回滚本轮，不叠加下一轮补丁。
