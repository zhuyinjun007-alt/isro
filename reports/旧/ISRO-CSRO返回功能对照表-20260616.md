## 1. 结论边界

- 本表只基于当前仓库中的既有报告、边界要求、修复成功记录和失败记录整理。

- 当前最重要的约束是：旧 CSRO 按钮必须保留原链路，新按钮只接入新增链路。

## 4. 旧链路与新链路的分界

### 必须保留的旧链路

- `0x21` 真实入口。
- CSRO 旧返回按钮的原 handler。
- 旧三功能 UI 的原语义。

### 只能新增的新链路

- `0xCC`：移动至队员位置。
- `0xCF`：移动至使用者指定的位置。
- 与这两个按钮直接相关的新客户端事件、GS 校验和传送分支。

### 明确禁止的路线

- 全局替换 message-map entries 指针。
- 把旧按钮统一接到新 handler。
- 再次以 `0x2A/0x2B` 当真实入口。
- 把静态字符串存在当作功能完成。

## 5. 当前需要补齐的映射

| 映射项 | 需要补齐的内容 | 说明 |
| --- | --- | --- |
| 按钮 ID → handler | `0xC8/0xC9/0xCA/0xCB/0xCC/0xCF` 的最终分派关系 | 先分旧/新，再谈合并 |
| handler → 协议包 | 旧功能与新增功能各自发往哪个包 | 不能混用 |
| 协议包 → GS 校验 | 是否需要组队、位置、白名单、对象状态校验 | 这是功能成立的关键 |
| UI → 生命周期 | 打开、关闭、第二次打开是否都稳定 | 这是当前高风险点 |

## 6. 现阶段结论

- ISRO / CSRO 的返回功能不能按“整体搬运”处理。
- 最小正确路线应是：旧按钮保持原链路，只给 `0xCC/0xCF` 追加新链路。
- 下一轮任何改动前，都要先冻结当前 hash 和备份状态，再做定点验证。
- 现有证据还不足以宣布“返回功能完整修复成功”。
