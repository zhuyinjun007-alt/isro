## 1. 总结结论

当前项目最大的问题不是“还差一个地址没找到”，而是开发过程曾经多次把局部静态证据当作完整成功证据：字符串存在、分支存在、静态脚本通过、界面能显示，都不能单独证明返回功能完整闭合。

截至本报告，本项目已有明显进展：

- 已确认真实右键返回卷轴入口是 `SetMsgBoxHandler arg0=0x21`，不是早期多轮围绕的 `0x2A/0x2B`。
- 已确认 CSRO 内部存在可复用的五/六按钮 UI 分支 `0x12E39EF`，ISRO 对应参考分支是 `0x722C4F`。
- 已经做出过“ISRO 样式 UI 显示层”的 Phase6 方案，关键是 `0x007BAAD9 -> 0x012EC930` field bridge stub，并在进入 `0x012DFA80` 前桥接 UI 字段。
- 当前深链检查 `deep_reverse_chain_check.py --targets --json` 显示 11/11 通过，说明 EXE/GS 中不少协议链路静态证据已经存在。

但不能称为完整成功：

- 用户记录显示 Phase6 后“ISRO 返回 UI 已出现”，但 CSRO 原返回功能全失效，点击“移动至使用者指定的位置”会客户端闪退。
- 本次新跑的 `static_reverse_check.py --targets --json` 仍有 3 个失败项，且集中在客户端入口、初始化/事件表路线和 message-map 指针口径上。
- 现有证据没有证明游戏内 6 个按钮全部可用，也没有证明右键第一次/第二次、旧功能、新功能、FatalLog/Dump 全部通过。

我的判断：后续不应继续在“整张事件表替换 + 全局 message-map 指针改写”的路线叠补。正确方向应改为“保留旧 CSRO 按钮原 handler，只为新增 `0xCC/0xCF` 接入新增链路；UI 显示可以借 Phase6 field bridge，但事件分发必须精细拆分”。

## 5. 根因级反思

### 5.1 问题一：把“界面问题”和“完整功能链路”混在同一个成功口径里

用户最终要的是：

- ISRO 返回证书/卷轴样式界面。
- CSRO 原有返回功能保持可用。
- 新增“移动至队员位置”。
- 新增“移动至使用者指定的位置”。
- EXE 与 GS 全链路闭合。
- 第一次、第二次右键都不闪退。

过去多轮开发里，部分阶段把 UI 显示当作成功推进依据。这会制造假进展：

- UI 能显示，只证明构造/布局走到了某条分支。
- 点击能进入 handler，只证明事件有分发。
- 能发包，只证明 EXE 侧产生了请求。
- GS 有 handler，只证明服务端有静态接收可能。
- 只有“游戏内旧功能 + 新功能 + 异常路径 + 重复打开 + 日志/Dump”都过，才接近完整成功。

后续必须把验收拆成 6 条独立门禁，不允许用其中一条替代全部。

### 5.2 问题二：真实入口曾长期判断错误

已读记录明确：

- 真实右键返回卷轴进入 `SetMsgBoxHandler` 时 `arg0=0x21`。
- 过去多轮围绕 `0x2A/0x2B/0x25/0x28` 修补，未命中真实 `0x21` 入口。
- `0x25 -> 0x12E39EF` 是五/六按钮分支，但它存在不等于右键返回卷轴会走到它。

这是典型的“二进制里有目标分支，但业务入口不可达”的问题。后续任何新补丁都必须先回答：

- 用户右键返回卷轴的真实 caller 是谁？
- 参数是什么？
- 跳转前后 `ECX/EDX/ESI` 语义是否保留？
- 这条入口是否能第二次打开并正常析构？

### 5.3 问题三：静态验证脚本被当成最终证明

Phase4 是最好的反例：

- 静态检查曾经全绿。
- 游戏内仍然右键闪退。
- dump 指向对象/映射/生命周期问题，而不是某个简单 opcode 缺失。

所以后续静态脚本的角色应调整为“防回归与证据采集”，不是最终裁判。静态脚本必须增加验证层级：

- 字节/跳转存在。
- 分支可达。
- 寄存器和对象字段语义正确。
- 旧 handler 是否保持原链路。
- 新 handler 是否只接新按钮。
- 游戏内手测与 FatalLog/Dump 证据。

### 5.4 问题四：全局替换 message-map 指针破坏了旧功能

当前最关键的失败点是 Phase6 的事件表策略。

已读记录显示：

- Phase6 把 `0x00EBE9F8` message-map entries 指针改到追加表 `0x012ED000`。
- 追加表不仅新增 `0xCC/0xCF`，还把旧按钮 `0xC8/0xC9/0xCA/0xCB` 等改成了新 handler。
- 结果是旧 CSRO 返回功能脱离原本 handler/协议链路，导致 UI 虽然显示，但旧功能不能用。

这说明 Phase6 的 UI field bridge 思路有价值，但“整张事件表替换”风险过大。后续应保留：

- `0x21` 真实入口。
- field bridge 中对 UI 关键字段的补齐。

后续应放弃：

- 全局替换 message-map entries 指针。
- 让旧按钮统一走 ISRO/new handler。
- 用新增表覆盖旧 CSRO 事件语义。

### 5.5 问题五：CSRO 与 ISRO 结构差异没有被系统化建模

用户一开始就强调：

- ISRO 和 CSRO 内存地址不同。
- GS 只能和 GS 对比，EXE 只能和 EXE 对比。
- 不能照搬地址。

但实际开发中，仍然出现过“看到 ISRO 同源分支 -> 在 CSRO 找相似分支 -> 直接接过去”的倾向。这里缺少一个结构映射表：

- ISRO 字段偏移与 CSRO 字段偏移一一对应关系。
- ISRO handler 与 CSRO handler 的语义对应关系。
- 旧 CSRO 按钮 ID 到旧 handler 的原始映射。
- 新增 ISRO 按钮 ID `0xCC/0xCF` 到新增 handler 的映射。
- 构造、显示、隐藏、析构涉及的对象字段。

没有这个映射表，补丁就容易变成“能跑到某段代码”而不是“语义等价”。

### 5.6 问题六：回滚边界有进步，但失败状态仍可能停留在 live 目标

项目中已经有大量 manifest 和 backups，这是好事。用户记录也显示多次定点回滚成功。

但当前 live hash 是 Phase6 后 hash，而用户记录显示 Phase6 实测旧功能失效/指定位置闪退。也就是说：

- manifest 机制存在。
- 备份机制存在。
- 但每轮失败后的“是否已经把 live 目标恢复到可继续实验的状态”需要单独确认。

后续每次实验后必须有状态决策：

- 如果游戏内失败，是否已按本轮备份恢复？
- 如果暂不恢复，是不是明确标记 live 目标处于失败实验态？
- 下一轮补丁的 pre_hash 是否就是这个失败态？

否则会在失败态上继续叠加，导致根因越来越难分离。

### 5.7 问题七：现有检查脚本有历史包袱

`static_reverse_check.py` 当前仍期望一些旧路线，例如：

- 某些旧 crash-safe type guard。
- 某些旧 `.rret` initializer。
- message-map 指针指向 `0x12EB600`。

但用户记录中又明确写过 `0x12EB600` 当前不是空洞，禁止再覆盖此地址作为事件表。说明脚本本身已经混入历史方案假设。

这会带来两个风险：

- 新正确路线可能被旧脚本误报失败。
- 旧失败路线可能被旧脚本鼓励复用。

建议后续把验证脚本分层重写：

- `check_baseline_hashes.py`：只看目标 hash 与备份状态。
- `check_ui_entry_runtime_route.py`：只看 `0x21` 真实入口与 field bridge。
- `check_old_button_handlers_preserved.py`：专门确认旧按钮仍走原 CSRO handler。
- `check_new_button_handlers_added.py`：专门确认 `0xCC/0xCF` 新增链路。
- `check_gs_protocol_routes.py`：只看 GS 协议与白名单/队伍校验。
- `check_final_acceptance_matrix.md`：记录游戏内人工验证，不让静态脚本冒充游戏结果。

## 7. 建议的后续路线

### 7.1 先做状态冻结

在继续修之前，先不要写字节。先做一份“当前 live 状态登记”：

- 当前 EXE hash：`4C3594684E434BD9EAE072D66A72F4595B38E2DB0E2B57F8840BC45CAE9DFC7A`
- 当前 GS hash：`D89E128FECD1BE9FE2E0D67102A407213CFDF9547D65845092BBAAB0EB57373A`
- 当前状态标签：Phase6 后；深链静态通过；旧功能/指定位置游戏内失败记录存在。

然后由用户确认：

- 是先恢复 Phase6 前备份，再重新做精细事件分发？
- 还是保留当前 UI 成功态，只在此基础上修旧 handler 保留问题？

从风险上看，我更倾向先恢复到 Phase6 前备份或至少复制出当前失败态备份，再做新实验。因为当前已经有旧功能失效证据，在失败态上继续叠补会污染根因。

### 7.2 保留 Phase6 的 field bridge 思路，但废弃整表替换

可保留：

- `0x007BAAD9` 真实右键 caller。
- `0x012EC930` field bridge stub 的方向。
- 对 `[ecx+0x810]` 等关键 UI 字段的桥接。
- 只处理 `0x21/0x25/0x28`，其它类型回原逻辑。

必须废弃：

- 全局替换 `0x00EBE9F8` entries 指针到整张追加表。
- 把旧按钮 `0xC8/0xC9/0xCA/0xCB` 改成新 handler。
- 让旧 CSRO 功能穿过 ISRO/new handler。

新方案应该是：

- 旧按钮 ID 继续走原 CSRO handler。
- 新按钮 `0xCC/0xCF` 才走新增 handler。
- 如果原 message-map 没有 `0xCC/0xCF`，优先考虑 wrapper dispatch：先判断是否 `0xCC/0xCF`，是则进新增 handler，否则原样转发到原 message-map。
- 不替换整张表，不迁移旧表项。

## 10. Code-audit 收口结论

审计深度：高风险。原因是任务涉及二进制补丁、客户端 UI 生命周期、网络协议、GS 校验、旧功能兼容和崩溃风险。

需求对账：

| 需求点 | 当前状态 | 证据 | 结论 |
| --- | --- | --- | --- |
| ISRO 样式返回 UI | 阶段性实现过 | Phase6 记录、field bridge manifest | 部分落地 |
| 保留 CSRO 原返回功能 | 当前证据显示失败 | `踩过的坑.md` Phase6 记录 | 未通过 |
| 新增移动至队员位置 | 静态链路存在 | deep check 11/11 pass | 证据不足，需游戏内验证 |
| 新增移动至使用者指定位置 | 静态链路存在，但用户记录点击闪退 | deep check pass + Phase6 失败记录 | 未通过 |
| GS/EXE 链路闭合 | 静态证据较完整 | deep check pass | 证据不足 |
| 第一次/第二次右键不闪退 | 当前无新鲜通过证据 | 用户记录曾闪退 | 证据不足/未通过 |
| 回滚边界 | manifest/backups 较完整 | evidence/backups | 部分可靠，仍需状态冻结 |

最终结论：有阻断风险，不可称完成。

阻断风险：

- 当前 live 目标可能处于 Phase6 失败实验态。
- 旧功能 handler 被全局事件表替换污染。
- 验证脚本口径冲突，不能直接用单个脚本作为推进依据。
- 缺少旧按钮 handler 保留表。
- 缺少游戏内完整验收矩阵结果。

建议的下一步最小动作：

1. 冻结当前 live hash 和备份。
2. 用户确认是否先恢复 Phase6 前备份。
3. 只读采证旧按钮 `0xC8/0xC9/0xCA/0xCB` 原 handler 与当前 handler 差异。
4. 设计 wrapper dispatch，只拦截新增 `0xCC/0xCF`，旧按钮回原链路。
5. 修改前写新检查脚本，先让“旧按钮保持原 handler”成为红灯测试。
6. 每轮写入后按验收矩阵进游戏验证，并记录 FatalLog/Dump。
