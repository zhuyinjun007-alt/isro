1. 保留当前已经做出来的 ISRO 样式返回 UI：5 个功能行 + 取消。
2. `移动至队员位置` 在单人/未满足 2 人组队条件时，必须像 ISRO 截图一样置灰不可选。

5. 不动其它补丁。

## 这几次失败的核心反思

### 1. UI 成功不等于功能成功

当前 UI 能显示，只能证明界面构造进入了某条可显示路径。它不能证明：

- 旧 CSRO 按钮仍走原 handler。
- 新按钮 handler 上下文完整。
- 列表请求、响应、选择、传送链路闭合。
- UI 关闭和第二次右键生命周期安全。

Phase9 的游戏内结果已经证明：静态全绿仍然会出现旧功能失效、新按钮点击闪退、队友按钮状态错误。

### 2. 旧按钮被污染是必须先修的主问题

用户当前最明确的红灯之一是“以前 CSRO 有的功能全没有了”。因此下一轮不能先追新功能，必须先恢复旧 CSRO 功能。

正确边界：

- `0xC8/0xC9/0xCA/0xCB` 必须保持原 CSRO 链路。
- 只有 `0xCC/0xCF` 可以进入新增链路。
- 如果旧按钮在任何表、wrapper、bridge 或运行时分发里被送进新 handler，本轮直接判失败。

### 3. 队友位置不是普通按钮，而是有状态按钮

`移动至队员位置` 必须有组队门槛：

- 单人或未满足 2 人组队：显示为 ISRO 式置灰，不可选择，不应发包，不应闪退。
- 2 人及以上组队：才允许选择，并继续走队员位置列表/传送链路。

这说明 EXE 侧必须提取 ISRO 的按钮状态判断链路，不能只补按钮文字或事件 ID。

### 4. 新按钮闪退优先怀疑 EXE UI 上下文，而不是直接归因 GS

已知 dump 形态是客户端 read AV，访问低地址，例如 `0x0000007C`。这更像 UI 对象、列表对象、上下文指针或生命周期字段未初始化。

除非日志证明请求已经成功发到 GS 并由 GS 处理，否则不能把新按钮闪退先归因给 GS。

### 5. `D:\Files\SR_GameServer.exe` 可以作为 CSRO GS 同结构参考，但不能替代 live 备份

用户说明 `D:\Files\SR_GameServer.exe` 与目标 GS 是同版本/同结构参考。它适合用于：

- 对比 live GS 中哪些字节属于返回相关改动。
- 排除 `_REFSKILL -> _REFSKILV` 这类非返回污染。
- 辅助恢复旧 CSRO GS 链路的语义。

但它不能替代每轮修复前的 live GS 备份。失败时仍必须恢复本轮备份的 `D:\CSRO\8-SR_GameServer.exe` 和 `F:\CSRO客户端\SRO_Client.exe`。

## 下次必须禁用的失败方式

1. 禁止继续把 `0x2A/0x2B` 当真实右键入口。
2. 禁止整张 message-map / entries 指针替换。
3. 禁止把旧按钮 `0xC8/0xC9/0xCA/0xCB` 改到新 handler。
4. 禁止只看 `0x12E39EF`、字符串、静态 pass 就当成功。
5. 禁止不先冻结当前 hash / 不先备份就写补丁。
6. 禁止不验证第一次和第二次右键。
7. 禁止不验证旧功能就继续加新功能。
8. 禁止在 Phase9 失败态上继续叠补丁。
9. 禁止为了证明 GS 被修改而写无意义字节。
10. 禁止点击新按钮闪退时不看 dump/上下文，直接猜 GS 逻辑。

## 下一轮正确顺序

1. 先读取 `每轮修复前必读.md`、`踩过的坑.md`、验收与回退清单、本报告。
2. 同时备份当前 EXE 和 GS。
3. 如果继续遵守失败回退铁律，先从 Phase9 本轮备份恢复 EXE/GS，再重新冻结 hash。
4. 保留 UI 成功方法，但拆掉/废弃会污染旧按钮的整表替换路线。
5. 先恢复旧 CSRO 返回功能，确认 `0xC8/0xC9/0xCA/0xCB` 走原链路。
6. 再修 `移动至队员位置` 的置灰/可用条件。
7. 最后只给 `0xCC/0xCF` 接新增链路。
8. 游戏内验收必须覆盖：第一次右键、第二次右键、旧功能、单人队友按钮置灰、2 人组队队友传送、指定位置按钮不闪退、FatalLog/Dump。

## 本轮设计倾向

推荐路线不是重做 UI，也不是整体搬 ISRO 分支，而是：

- 保留当前 ISRO 样式 UI 外观资产。
- 旧按钮回原 CSRO 链路。
- `移动至队员位置` 先只接状态判断，让单人置灰。
- `0xCC/0xCF` 新链路必须从 ISRO EXE/GS 一一对比提取，再按 CSRO 结构重建。

在旧功能恢复前，不进入新功能点击链路修复。
