## 已确认失败点

- live EXE 仍是整表替换失败态：`0x00EBE9F8 -> 0x012ED000`。
- 这会让旧 CSRO 按钮脱离原始 entries/runtime slot 组合，污染旧返回功能。
- 继续把 `0x012ED000` 当整张 message-map entries 表使用，是本轮明确禁用的失败方式。
- 只看 UI 文案、`0x12E39EF` 或静态 handler 存在不能判定成功。

## 本轮正确做法

只做小扩展，不整表替换：

| VA | 旧值 | 新值 | 目的 |
| --- | --- | --- | --- |
| `0x00EBE9F4` | `0x00EC21B4` | `0x012ED000` | base 指向小扩展 map |
| `0x00EBE9F8` | `0x012ED000` | `0x00FFE1E8` | entries 恢复原 CSRO 表 |
| `0x012ED000` | 整表失败态 | `{base=0x00EC21B4, entries=0x012ED020}` | 只扩展新按钮 |
| `0x012ED020` | 无有效小表 | `0xCC -> 0x012DEF50`, `0xCF -> 0x012DF020` | 只接新增功能 |

旧按钮保留方式：

- `0xC8/0xC9/0xCA/0xCB` 不进入 `0x012ED020`。
- 原 runtime initializer 继续写回：
  - `0xFFE228 -> 0x703F10`
  - `0xFFE250 -> 0x7040E0`
  - `0xFFE278 -> 0x70AB50`
  - `0xFFE2A0 -> 0x7041D0`

## GS处理

本轮同时备份了 GS，但没有做无意义 GS 字节写入。

原因：当前 GS 已静态闭合必要服务端链路：

- vtable slot 指向 `0x00E56000`
- handler page 包含 `0x3213/0x705A/0x759F/0x7600/B59F/B600`
- 包含返回物品白名单 `9114/24298`
- 包含队伍人数至少 2 人门槛
- 未处理消息 fallback 到原 `0x004C8410`

这只能说明 GS 当前静态链路存在，不能单独宣称游戏内最终成功。禁止为了让 LastWriteTime 变化而假写 GS。
