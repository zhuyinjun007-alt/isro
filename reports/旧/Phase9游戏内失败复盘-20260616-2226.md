## 失败原因复盘

1. Phase9 仍然只证明了静态表结构：`0x21`、UI bridge、小扩展表、`0xCC/0xCF` handler 静态存在。
2. 游戏内结果证明 UI 状态没有闭合：
   - 单人状态下 `移动至队员位置` 没有按 ISRO 置灰。
   - 点击新增按钮会进坏对象/坏上下文，产生 read AV。
3. 旧 CSRO 功能失效说明旧按钮虽然静态上看似保留，但运行时功能链路仍被 UI bridge、状态字段或 handler 上下文破坏。
4. GS 侧没有形成可信“同轮新增修复”证据；更不允许用 `_REFSKILV` 这种无关字节变化冒充 GS 修复。

## 禁止下次复用的方式

- 禁止继续在当前失败态上叠补丁。
- 禁止把 Phase9 静态全绿当作成功态继续推进。
- 禁止把 `0x12ED000` 重新当成整张 message-map entries 表。
- 禁止把旧按钮 `0xC8/0xC9/0xCA/0xCB` 接到新 handler。
- 禁止只看 `0x12E39EF`、`0xCC/0xCF` 字符串、handler 存在就宣布成功。
- 禁止为了 GS 时间变化写无意义字节。
- 禁止忽略第一次和第二次右键、旧功能、单人无队伍状态、新按钮点击这四类游戏内验收。

## 下一轮必须先做

1. 先保留最新 dump/FatalLog 证据。
2. 直接恢复本轮失败前备份中的 EXE 和 GS，不做手工补丁回退。
3. 恢复后重新冻结 EXE/GS hash、长度、LastWriteTime。
4. 重新对照 ISRO EXE/GS：
   - EXE：UI 创建、按钮 enable/disable 状态、旧按钮 handler、新按钮 handler、列表请求/响应、最终发包、关闭/第二次打开生命周期。
   - GS：`0x759F/0x7600` 列表请求、`0xB59F/0xB600` 响应、`0x3213` 队员传送、`0x705A type=5` 指定位置传送、物品白名单、组队人数门槛。
5. 先修复“旧 CSRO 三个功能恢复”和“单人状态下队员位置置灰”，再尝试新增两个功能点击链路。
