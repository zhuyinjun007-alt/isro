# 边界要求

整理时间：2026-06-16

## 1. 修复范围

- 只修 CSRO 的 GS 和 EXE：
  - GS：`D:\CSRO\8-SR_GameServer.exe`
  - EXE：`F:\CSRO客户端\SRO_Client.exe`
- 客户端实际启动入口：`F:\CSRO客户端\开始游戏.exe`。
- 不修、不动、不当作修复证据：
  - DB、PK2、Media 资源、DLL、launcher、ShardManager。
  - 强化、界限突破石、防爆石、药化、过滤器等其它补丁。

## 2. 参考对象

- 本轮 ISRO 参考 GS：`F:\SR_GameServer.exe`。
- ISRO 参考 EXE：`F:\ISRO客户端国服\sro_client.exe`。
- `D:\1SRO_DevKit - Bancroft` 只用于理解返回插件逻辑，不能当作最终补丁来源照搬。

## 3. 必须实现的功能

- CSRO 已有 4 个返回功能，只添加缺少的 2 个：
  - `移动至队员位置`
  - `移动至使用者指定的位置`
- 目标 UI 必须与 ISRO 附图一致：保留原有 3 个功能行，新增上述 2 个功能行，最后仍有 `取消`，即 5 个功能行 + 1 个取消行。
- `移动至队员位置` 必须至少 2 人组队才能传送。
- 右键 `返回卷轴` 第一次、第二次都不能闪退。
- UI 必须走 ISRO 完整链路，不能只补文字、按钮资源或静态分支。

## 4. 对比规则

- ISRO 和 CSRO 版本不同，内存地址不同，禁止直接套地址。
- GS 只和 GS 对比，EXE 只和 EXE 对比。
- 必须完整提取并闭合：
  - UI 创建
  - message-map / 事件
  - 发包
  - GS 收包、校验、传送
  - UI 关闭、析构、第二次打开生命周期
- 不确定时必须补证据，禁止用推测替代一一对比。

## 5. 数据库和资源定位

- 物品只作理解返回工作原理：
  - `9114` / `ITEM_QNO_WC_FERRY2_1_01`
  - `24298` / `ITEM_EVENT_REVERSE_RETURN_SCROLL`
- `_RefObjCommon`、`_RefObjItem` 及资源文本能说明物品存在，但不能证明 GS/EXE 功能已修好。
- 客户端资源补丁已打，剩余问题必须在 GS 和 EXE 解决。

## 6. 回滚铁律

- 第 10 条新增要求：修复之前必须先备份 GS 和 EXE；如果修复失败，必须直接还原本轮备份的 GS 和 EXE；此“备份-写入-验证-失败还原”流程必须在每一轮修复中循环使用。
- 修复前必须先备份当前 CSRO GS 和 EXE：
  - GS：`D:\CSRO\8-SR_GameServer.exe`
  - EXE：`F:\CSRO客户端\SRO_Client.exe`
- 每一轮写入前都要生成本轮独立备份，并记录 hash、备份路径、写入 manifest。
- 如果本轮修复失败，必须直接还原本轮备份的 GS 和 EXE，再进入下一轮修复；这个备份-写入-验证-失败还原流程必须循环使用。
- 每次失败只回滚本轮失败的返回补丁字节。
- 已验证成功或与返回无关的补丁必须保留。
- 禁止为了排错整文件覆盖 GS 或 EXE 到最初状态。
- 如果证据不能区分失败来源，状态设为“证据不足”，先查日志、dump、运行时链路，再动字节。

## 7. 验证门禁

- 每次修复前必须先阅读：
  - `D:\做ISRO返回记录\每轮修复前必读.md`
  - `D:\做ISRO返回记录\踩过的坑.md`
  - `D:\做ISRO返回记录\reports\返回功能验收与回退清单-20260616.md`
- 没有修改 GS 或 EXE，不需要进游戏验证，结果必然一样。
- 修改后必须记录：
  - GS/EXE hash
  - VA / file offset / bytes
  - 静态链路检查
  - 游戏内实测时间点
  - `D:\CSRO\FatalLog` 同时间段日志
  - `F:\CSRO客户端\Dump` 是否新增 dump
- 静态通过只算证据之一，不算最终修复成功。
