# ISRO 新增返回功能严格对齐设计

## 目标

本设计只面向 CSRO 返回卷轴中缺少的两个 ISRO 返回功能：

- `移动至队员位置`
- `移动至使用者指定的位置`

目标不是先做一个“能点、能传送”的低配版本，而是尽量严格对齐 ISRO 行为。客户端按钮状态、点击事件、发包、服务端收包、服务端校验、传送执行、失败拒绝、UI 关闭和第二次打开都必须闭合。

最终验收以游戏内复现为准。静态检查只能作为准入证据，不能单独证明修复成功。

## 范围

允许修改：

- `F:\CSRO客户端\SRO_Client.exe`
- `D:\CSRO\8-SR_GameServer.exe`

只读参考：

- `F:\ISRO客户端国服\sro_client.exe`
- `F:\SR_GameServer.exe`
- `D:\1SRO_DevKit - Bancroft`

`D:\1SRO_DevKit - Bancroft` 只能帮助理解返回插件逻辑，不能作为最终补丁来源照搬。最终行为必须以 ISRO EXE 和 ISRO GS 的真实链路为标准，再映射到 CSRO 的实际结构。

不在本设计范围内：

- DB、PK2、Media 资源、DLL、launcher、ShardManager
- 强化、界限突破石、防爆石、药化、过滤器等无关补丁
- 整文件覆盖 live EXE 或 live GS
- 复用已验证失败的整张 message-map 替换路线

## 必须保留的边界

- 真实右键返回卷轴入口以 `0x21` 为准。
- 旧 CSRO 按钮 `0xC8/0xC9/0xCA/0xCB` 必须保持原 CSRO 链路。
- 新链路只允许接 `0xCC/0xCF`。
- `移动至队员位置` 在单人未组队时必须灰色不可选，至少两人组队才允许使用。
- 第一次右键和第二次右键返回卷轴都不能导致客户端闪退。
- 每轮写入前必须同时备份 EXE 和 GS。失败后直接恢复本轮备份，不在失败态叠补丁。
- 当前 `_REFSKILV` 是用户独立补丁，不属于返回功能失败证据，后续不得清理或回滚。

## 设计方案

采用三层闭合方案，逐层推进。前一层没有通过运行时验证，不进入后一层写入。

### 第一层：客户端入口与旧链路稳定

目标是先锁定客户端基础状态，不急着实现新增传送。

本层要确认：

- `0x21` 右键返回卷轴仍打开 ISRO 风格返回 UI。
- 旧按钮 `0xC8/0xC9/0xCA/0xCB` 没有进入新增 handler。
- 第一次和第二次打开返回 UI 生命周期稳定。
- `0xCC` 的可选状态来自真实运行时队伍/队员状态，而不是之前失败的猜测字段。
- 单人未组队时 `0xCC` 必须灰色不可选。

本层禁止：

- 整张替换 message-map entries。
- 把旧按钮塞进新增表。
- 只看字符串、分支存在或静态同构就判定成功。
- 复用 `state_global+0x30 >= 2` 或“只恢复 call 到 `0x00A7BE50`”这类已失败路线。

### 第二层：客户端新增事件与发包

目标是只给 `0xCC` 和 `0xCF` 建立最小新增事件链。

`0xCC` 链路：

1. 返回 UI 打开后，根据真实队伍状态决定按钮是否可选。
2. 玩家点击 `0xCC`。
3. 客户端进入新增 handler。
4. 客户端构造并发送 ISRO 语义对应的队员位置请求或传送请求。
5. 旧按钮分派不受影响。

`0xCF` 链路：

1. 返回 UI 打开后，玩家点击 `0xCF`。
2. 客户端进入新增 handler。
3. 客户端请求或展示使用者指定位置相关数据。
4. 玩家选择位置后发送 ISRO 语义对应请求。
5. 点击、选择、关闭和第二次打开都不能导致客户端 dump。

本层必须使用静态对比和运行时观察互相校验。若静态证据与运行时证据冲突，以运行时证据为准。

### 第三层：GS 校验与传送行为

目标是让 GS 对两个新增请求具备 ISRO 对齐的服务端语义。

GS 需要闭合：

- 收包识别
- 返回物品校验
- 角色状态校验
- 队伍与队员目标校验
- 使用者指定位置校验
- 地图、副本、职业状态、传送限制校验
- 传送执行
- 失败安全拒绝
- 日志与 FatalLog 观察

服务端必须重新校验关键条件，不能只信客户端按钮置灰。客户端按钮状态只改善交互体验，不是安全边界。

## 数据流

### 移动至队员位置

1. 客户端通过 `0x21` 打开返回卷轴 UI。
2. 客户端读取真实运行时队伍/队员状态。
3. 单人未组队时 `0xCC` 灰色不可选。
4. 两人及以上组队时 `0xCC` 可选。
5. 玩家点击 `0xCC` 后，客户端进入新增 handler 并发出 ISRO 语义对应请求。
6. GS 收包后校验物品、角色状态、队伍状态、目标队员、地图/副本/职业限制和传送可达性。
7. 校验通过才执行传送。
8. 校验失败必须安全拒绝，不产生客户端 dump 或 GS FatalLog。

### 移动至使用者指定的位置

1. 客户端通过 `0x21` 打开返回卷轴 UI。
2. 玩家点击 `0xCF`。
3. 客户端进入新增 handler，并请求或展示使用者指定位置数据。
4. 玩家选择位置后发出 ISRO 语义对应请求。
5. GS 收包后校验物品、保存位置、角色状态、地图/副本/职业限制和传送目标。
6. 校验通过才执行传送。
7. 校验失败必须安全拒绝，不重复扣物品、不重复传送、不产生 FatalLog。

## 证据与调试策略

本设计允许使用动态观察，包括启动服务/客户端、读取日志、保留 dump、断点和运行时观察。动态动作必须限定在授权本机环境内，目的只用于确认返回功能链路。

每个结论至少绑定一种证据：

- EXE/GS hash、大小、时间戳
- VA、raw offset、old bytes、new bytes
- 静态调用链或分发表证据
- 运行时参数、断点记录或日志
- 客户端 dump 路径和异常地址
- GS FatalLog 时间段
- 游戏内截图或复现记录

没有证据的推断只能写为假设，不能作为写入依据。

## 每轮执行流程

每一轮修改必须小步执行，一次只验证一个明确闭合点。

执行前：

1. 冻结当前 EXE/GS hash、时间和大小。
2. 确认 EXE/GS 未被运行进程锁住。
3. 同时备份 `SRO_Client.exe` 和 `8-SR_GameServer.exe`。
4. 生成 manifest，记录 VA、raw offset、old bytes、new bytes 和修改理由。

执行中：

1. 写入最小补丁。
2. 运行静态检查。
3. 必要时执行动态观察。
4. 若出现客户端 dump 或 GS FatalLog，先保留证据，再回滚。

执行后：

1. 复核 EXE/GS hash 和 LastWriteTime。
2. 游戏内验证第一次右键、第二次右键、旧功能、新功能和无新增 dump/FatalLog。
3. 将结果写入 `reports/`。
4. 关键推进或阻塞时回复 GitHub issue。

失败时：

1. 直接恢复本轮备份的 EXE 和 GS。
2. 复核 hash 回到备份值。
3. 记录失败证据。
4. 不在失败态继续叠补丁。

## 测试与验收

验收分三层，不能跳级。

### 第一层：客户端稳定性验收

- 第一次右键返回卷轴 UI 正常出现。
- 关闭后第二次右键仍正常出现。
- 旧 `0xC8/0xC9/0xCA/0xCB` 功能仍可用。
- 单人未组队时 `移动至队员位置` 灰色不可选。
- 无新增客户端 dump。

### 第二层：新增按钮链路验收

- `0xCC` 在两人及以上组队时变为可选。
- `0xCC` 点击后不闪退，并进入正确队友目标或传送链路。
- `0xCF` 点击后不闪退，并进入使用者指定位置链路。
- 客户端发包语义与 ISRO 对应行为一致。

### 第三层：GS 完整行为验收

- GS 能收到并识别两个新增请求。
- GS 按 ISRO 规则校验物品、队伍、角色状态、地图/副本/职业限制和目标合法性。
- 成功场景能传送。
- 失败场景安全拒绝。
- 不重复扣物品。
- 不重复传送。
- 无新增 GS FatalLog。

## 交付物

每个实际修复轮次必须交付：

- 本轮备份目录
- patch manifest
- 静态检查输出
- 动态观察或游戏内验证记录
- dump/FatalLog 复核结果
- `reports/` 下的阶段记录
- 必要时的 GitHub issue comment

## 结论

本项目可继续推进，但不能走“整体搬运 ISRO”或“先做能点能传送”的路线。正确路线是分层闭合：先稳定客户端入口与旧链路，再只为 `0xCC/0xCF` 补新增事件和发包，最后按 ISRO 语义补 GS 校验与传送。

该方案比最小可用路线更慢，但能避免已经多次出现的旧按钮污染、UI 生命周期崩溃、单人置灰失败和 EXE-only/GS-only 伪成功问题。
