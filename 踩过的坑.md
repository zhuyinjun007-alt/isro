# 踩过的坑

整理时间：2026-06-16

## 1. 真实入口判断错

- 已证实真实右键返回卷轴进入 `SetMsgBoxHandler` 时 `arg0=33`，即 `0x21`。
- CSRO 运行分支：
  - `0x21 -> 0x12E3464`，实际三选项分支。
  - `0x25 -> 0x12E39EF`，五选项分支。
  - `0x2A -> 0x12E489C`
  - `0x2B/0x2C/0x2D -> 0x12E1962`
- 之前多轮围绕 `0x2A/0x2B/0x25/0x28` 修改，未命中真实 `0x21` 入口，所以界面没有变化。

## 2. 只看静态会误判

- 静态脚本绿，只能证明某些跳转、表项、字节存在。
- 它不能证明右键返回卷轴真实会走到五选项分支。
- 后续必须同时用运行时参数、游戏内 UI、dump、FatalLog 交叉验证。

## 3. CSRO 有五选项分支，但不可达

- CSRO 五选项分支 `0x12E39EF` 存在。
- ISRO 五选项参考分支是 `0x722C4F`。
- 按 jump table 分支边界截断后，CSRO/ISRO 的 `0x21` 都是旧三功能 UI，不含 `0xCC/0xCF` 新增按钮；CSRO `0x25 -> 0x12E39EF` 与 ISRO `0x25 -> 0x722C4F` 才是附图 2 新增两按钮 UI 的同源分支。
- 分支中能看到：
  - `UIIT_CLT_REVERSE_RETRUN_SCROLL_PARTY_MOVE`
  - `UIIT_CLT_REVERSE_RETRUN_SCROLL_USER_APPOINT`
  - `UIIT_CLT_REVERSE_RETRUN_SCROLL_TO_APPOINT_CHANGE`
- 错误点：把“字符串/分支存在”当成“UI 已接入”。实际入口、事件、发包、析构没有闭合。
- 另一个错误点：用过大的反汇编窗口扫描 `0x21`，会扫进后面的 `0x25` 分支并误判 `0x21` 已有新按钮；后续必须按 jump table 的下一个分支起点确定扫描范围。

## 4. UI 生命周期路线已失败，禁止复用

已失败路线：

- v10：按钮框出现，但第二次右键返回卷轴崩溃。
- v11：避开 `[esi+0x818]/[esi+0x81C]` 后仍崩。
- v12：只补点击事件 bridge，未解决 allocator/list 根因。
- v13：补 `EDI=8` 后仍失败，并影响旧返回和进游戏链路。

失败 dump 形态：

- `0x00425C3F` read AV
- `0x00902E4A` write to `0x00000000`
- `0x003DCCCC` / 无效 freelist 指针

结论：崩溃不是崩溃点字节坏了，而是新增返回 UI 创建、挂载、销毁污染了客户端 UI allocator/list 或对象生命周期。

## 5. 明确禁止再用的方法

- `0x711770 -> 0x12EB600`
- `CIFMessageBox` 扩容
- message-map/static-init 猜迁移
- 猜写 `[esi+0x818]` / `[esi+0x81C]`
- 单独补 button skin
- 单独补 click bridge
- 单独补 `EDI=8`
- Round6/Round9/Round10/Round11/Round12 这种继续绕 `0x2A/0x2B` 的方法

补充记录（2026-06-16）：

- 旧 `static_reverse_check.py` 的 5 个失败项要求旧路线：`SetMsgBoxHandler` type guard、`0x2A/0x2B -> 0x25`、static initializer 改到 `.rret`、`0x12EB600` 事件表和 lifecycle guard。
- 这套旧路线与前面失败记录重合，不能作为继续修复依据。
- 只允许在一一对比确认后，补“可达事件表缺少 `0xCC/0xCF`”这种最小闭合点；不得再改 static initializer，不得再改 lifecycle guard，不得再把 `0x2A/0x2B` 当真实入口。
- `0x12EB600` 当前不是空洞，含已有数据，禁止再覆盖此地址作为事件表。

## 6. Round10 / Round11 / Round12 不算成功

- Round10：加了 `[esi+0x810]` null-safe guard，但没有命中真实 `0x21` UI 入口。
- Round11：把两处静态入口 `push 0x2A` 改成 `push 0x25`，仍没解决真实运行分支。
- Round12：把 `0x2A/0x2B` 映射到 `0x25`，用户实测界面无变化，已回滚。
- 这些记录从“修复成功过的”删除，只保留为失败经验。

## 7. GS-only / EXE-only 都不完整

- 只改 EXE：UI/发包可能出现，但 GS 接收、校验、传送不完整。
- 只改 GS：客户端 UI 和请求链不走 ISRO 功能，游戏内仍看不到两个新增选项。
- 完整修复必须 GS 和 EXE 同时闭合。

## 8. DB/资源不是成功证据

- `ITEM_EVENT_REVERSE_RETURN_SCROLL`、`ITEM_QNO_WC_FERRY2_1_01`、`UIIT_*` 在资源或数据库中存在，只说明物品/文本存在。
- D 盘 SR_GameRefData、PK2 文本不能作为 GS/EXE 修复成功证据。

## 9. 误判入口要排除

- 类似 `0x746331`、`0x7464C4` 的路径曾被误判，但不属于返回卷轴五选项链路。
- 以后必须用字符串、类型、调用方、运行时参数一起确认，不能只看 message box 形态相似。

## 10. 整文件恢复会误伤其它补丁

- `D:\SR_GameServer.exe`、`I:\SRO_Client.exe` 是原始参考，不是直接覆盖目标。
- 当前 live GS/EXE 可能有其它已保留补丁。
- 正确回滚方式：按 manifest 的 raw offset / VA / bytes 只恢复失败返回补丁。

## 2026-06-16 15:35 - UI 成功但旧功能失效/指定位置闪退的失败点

本轮用户实测结果：

- ISRO 返回 UI 已出现。
- CSRO 原来的返回功能全失效。
- 点击 `移动至使用者指定的位置` 客户端闪退，dump：`F:\CSRO客户端\Dump\1N[2026-06-16 15-27-29]_88_34 谁与争锋[西南]_ak48.dmp`。

已确认失败原因：

- Phase6 把 `0x00EBE9F8` message-map entries 指针改到追加表 `0x012ED000`。
- 追加表不仅新增 `0xCC/0xCF`，还把旧按钮 `0xC8/0xC9/0xCA/0xCB` 等也改成了新 handler，例如：
  - `0xC8 -> 0x012DEB20`
  - `0xC9 -> 0x012DE6E0`
  - `0xCC -> 0x012DEF50`
  - `0xCF -> 0x012DF020`
- 这会让旧 CSRO 功能脱离原来的 CSRO handler/协议链路，所以界面虽然像 ISRO，但旧功能不能用。

后续禁止复用的方法：

- 禁止把旧 CSRO 返回按钮统一接到 ISRO/new handler。
- 禁止用“追加整张事件表 + 全局替换 message-map 指针”当作成功修复依据。
- 禁止继续沿用“旧按钮也走 `0x012DEB20/0x012DE6E0` 等新链路”的方案。
- 新修复必须做到：UI 可以是 ISRO 样式，但旧按钮保持 CSRO 原始功能链路；只允许新增的 `移动至队员位置` 和 `移动至使用者指定的位置` 进入新增链路。
